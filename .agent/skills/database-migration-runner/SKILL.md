# Skill: Database Migration Runner

# Purpose

This skill standardizes:
- Prisma schema updates
- database migrations
- auth model evolution
- database safety practices

inside SecureGate.

Authentication systems depend heavily on database correctness.

A broken migration can:
- destroy auth data
- invalidate sessions
- break verification flows
- corrupt token systems
- create production deployment failures

This skill ensures database changes remain:
- predictable
- reversible
- production-safe
- aligned with the SecureGate architecture

Use this skill whenever:
- modifying Prisma schema
- creating migrations
- changing auth models
- updating token structures
- evolving session storage

---

# SecureGate Database Philosophy

SecureGate uses:
- PostgreSQL
- Prisma ORM

The database architecture must remain:
- explicit
- predictable
- minimal
- security-focused

Avoid:
- over-normalization
- unnecessary relations
- experimental patterns
- premature abstractions

Authentication systems should prioritize:
- clarity
- reliability
- auditability

---

# Database Source Of Truth

The ONLY schema source of truth is:

```txt
prisma/schema.prisma
```

Never:
- create parallel schema systems
- manually drift production schema
- modify DB structure outside migrations

All schema changes must originate from Prisma.

---

# Prisma Client Rules

The ONLY Prisma client entry point is:

```txt
lib/prisma.ts
```

Never instantiate:

```ts
new PrismaClient()
```

outside the singleton.

Multiple Prisma clients cause:
- connection exhaustion
- development instability
- inconsistent behaviour

---

# SecureGate Database Models

SecureGate intentionally uses a small auth-focused schema.

Required models:

- User
- VerificationToken
- PasswordResetToken

Optional:
- Session (depending on auth strategy)

---

# Canonical User Model

```prisma
model User {
  id              String   @id @default(cuid())
  name            String
  email           String   @unique
  password        String
  emailVerified   Boolean  @default(false)
  createdAt       DateTime @default(now())
}
```

---

# Canonical VerificationToken Model

```prisma
model VerificationToken {
  identifier String
  token      String   @unique
  expires    DateTime
}
```

---

# Canonical PasswordResetToken Model

```prisma
model PasswordResetToken {
  email   String
  token   String   @unique
  expires DateTime
}
```

# Before You Modify The Schema

# Step 1. Read Context First

Before changing schema:
1. `architecture.md`
2. `security.md`
3. `AGENTS.md`

Understand:
- token lifecycle
- auth architecture
- session strategy
- verification behaviour

Never modify schema blindly.

---

# Step 2. Validate The Need

Before adding fields/models ask:
- Is this required by the assessment?
- Does this violate YAGNI?
- Does this improve auth reliability?
- Is this necessary for security?

Do NOT add:
- analytics tables
- role systems
- admin models
- social auth models
- audit systems
- unrelated SaaS structures

Keep scope intentionally narrow.

---

# Migration Workflow

# Step 3. Modify The Prisma Schema

Update:
```txt
prisma/schema.prisma
```

Keep schema:
- readable
- explicit
- predictable

Avoid:
- deeply nested relations
- unnecessary optionality
- unclear naming

---

# Naming Rules

## Models

Use:
- singular
- PascalCase

Good:
```prisma
User
VerificationToken
PasswordResetToken
```

Bad:
```prisma
Users
password_reset_tokens
```

---

## Fields

Use:
- camelCase

Good:
```prisma
emailVerified
createdAt
```

Bad:
```prisma
email_verified
created_at
```

---

# Step 4. Generate Migration

Use:

```bash
npx prisma migrate dev --name meaningful_name
```

Migration names should describe:
- what changed
- why it changed

Good:
```bash
add_password_reset_tokens
```

Bad:
```bash
fix_stuff
```

---

# Step 5. Inspect The Generated SQL

Before applying:
- inspect generated migration SQL
- verify indexes
- verify constraints
- verify deletes
- verify nullable fields

Never trust autogenerated migrations blindly.

---

# Step 6. Run Prisma Generate

After migrations:

```bash
npx prisma generate
```

Ensure Prisma types remain synchronized.

---

# Step 7. Verify Auth Flows Still Work

After migration verify:
- signup
- login
- verification
- password reset
- protected routes

Authentication systems break easily after schema changes.

---

# Security Rules

# Password Security

Passwords must:
- remain hashed
- never be stored in plain text
- never be exposed in responses

Database schema must never weaken password protections.

---

# Token Security

Verification and reset tokens must:
- expire automatically
- remain single-use
- invalidate after usage
- use unique constraints

Never allow:
- token reuse
- non-expiring tokens
- duplicated token rows

---

# Email Security

Emails must remain:
- unique
- indexed
- validated before persistence

Use:
```prisma
@unique
```

for user email fields.

---

# Forbidden Database Practices

DO NOT:
- use raw SQL unnecessarily
- use `$queryRawUnsafe`
- disable constraints
- remove uniqueness protections
- bypass migrations
- mutate production schema manually

---

# Transaction Rules

Use transactions for multi-step auth writes.

Examples:
- user creation + token creation
- password reset + token deletion
- verification + token invalidation

Use:

```ts
prisma.$transaction()
```

Never allow partial auth writes.

---

# Deletion Rules

Sensitive auth data should be cleaned up safely.

Delete:
- used verification tokens
- used reset tokens
- expired tokens

Never allow token buildup indefinitely.

---

# Migration Safety Rules

Before applying migrations:
- backup production data
- inspect destructive changes
- verify constraints
- verify indexes

Never casually:
- drop columns
- rename critical auth fields
- remove unique constraints

---

# Development vs Production

## Development

Use:
```bash
npx prisma migrate dev
```

---

## Production

Use:
```bash
npx prisma migrate deploy
```

Never use:
```bash
prisma db push
```

in production environments.

---

# Prisma Formatting Rules

Always run:

```bash
npx prisma format
```

after schema updates.

Keep schema readable and standardized.

---

# Indexing Rules

Index:
- auth lookup fields
- email fields
- token fields

Authentication systems depend heavily on fast lookups.

---

# Nullable Field Rules

Avoid nullable auth fields unless necessary.

Authentication systems become unstable when critical fields are optional.

Good:
```prisma
email String @unique
```

Bad:
```prisma
email String?
```

---

- [ ] Password reset still works
- [ ] Tokens invalidate correctly

- Defensive Programming

# Recovery Rules

If a migration fails:
1. Stop immediately
2. Diagnose the SQL
3. Revert safely
4. Fix schema intentionally
5. Re-run migration carefully

Never:
- manually patch production tables
- skip migration history
- ignore failed migrations

---

# What Not To Do

- Do not add unnecessary models
- Do not bypass Prisma
- Do not use raw SQL recklessly
- Do not store plain text passwords
- Do not create non-expiring tokens
- Do not duplicate Prisma clients
- Do not weaken unique constraints
- Do not use `db push` in production
- Do not over-engineer the schema

SecureGate database architecture should remain:
- minimal
- secure
- predictable
- production-ready
